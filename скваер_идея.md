# Autonomous Agent System — Binance Square

## Концепция

SDK = руки. Агент = мозг. **Runtime = нервная система.**

Сейчас агент — это LLM с промптом, который вызывает SDK. Между ними ничего. Лимиты в промпте, паузы в промпте, circuit breaker в промпте. Всё что в промпте — опционально, забываемо, ненадёжно.

Цель: 5+ полностью автономных агентов, каждый со своей персоной, нишей, стратегией. Работают по расписанию без вмешательства человека. Каждый стремится стать крупным инфлюенсером на Binance Square.

**Агенты не знают друг о друге.** Каждый — независимая личность с уникальным характером, стилем, мышлением. Как 5 реальных людей которые не знакомы, но все хотят стать инфлюенсерами. Уникальность обеспечивается не координацией, а identity: если персона прописана хорошо — два агента напишут про один и тот же дамп BTC по-разному, как два живых человека.

Разница между ботом и агентом: **обратная связь**. Бот повторяет сценарий. Агент видит результат → меняет подход. Без обратной связи любой агент через неделю становится ботом.

## Проблемы текущей системы (из реального тестирования)

Сессия 3 агента aisama обнажила цепочку проблем, каждая из которых усиливает следующую:

### Проблема 1: Бесцельный скроллинг
Фид оказался мусорным (все 20 постов с 0 лайков — SIGN спам). Агент добросовестно листал всё подряд, потому что промпт говорит "browse the feed naturally". Нет минимального порога качества, нет fallback когда фид — мусор.

### Проблема 2: Подготовка и публикация не разделены
Агент подготовил скриншот графика для поста. Пост не получился (timeout). Скриншот пошёл в мусор. Нет логики "если пост не получился с картинкой — адаптируйся".

### Проблема 3: Circuit breaker слишком грубый
`create_post()` упал 2 раза → агент решил "post creation broken this session" и прекратил ВСЕ попытки. Но комменты работают через другой DOM-путь. Один сломанный тип действия потянул за собой всю сессию.

### Проблема 4: Cascading failure
После 2 failures на пост + 1 failure на коммент, агент решил "браузер сломан" и сдался. Хотя SDK изолирует действия — каждое в своём connect/disconnect. Агент не знает об этой изоляции.

### Проблема 5: Journal врёт
Записано "7/8 successful". Реально: 5 минут действий и 35 минут бесполезного скроллинга. Journal фиксирует "что сделал", но не КПД. Оператор думает что всё хорошо, а сессия потеряна.

### Проблема 6: Нет плана до начала действий
Промпт говорит "Think about what's happening". Это размышления, не план. Нет конкретного deliverable list: "1 пост на тему X, 2 коммента на посты с 20+ лайков". Агент "думает" и идёт "скроллить".

### Цепочка
```
Мусорный фид (0 likes)
  → Нет фильтра → агент листает всё подряд
  → Нет плана → "browse naturally" = бесцельный скроллинг
  → Попытка поста → timeout → retry без паузы → timeout
  → Circuit breaker (2 fails) → "post creation broken"
  → Попытка коммента → fail → "browser broken"
  → Агент сдаётся, сессия потеряна
  → Journal записывает успехи, не записывает потраченное время
```

**Все эти проблемы решаются архитектурой ниже: planner даёт конкретный план ДО действий, guard контролирует circuit breaker КОДОМ по типам действий, collector/scorer измеряют реальный КПД.**

## Слой 1: Runtime — программные гарантии

**Принцип:** всё что агент НЕ должен решать сам — обеспечивается кодом. LLM не считает, не валидирует, не контролирует своё поведение.

### guard.py — ActionGuard

Каждое действие агента проходит через guard перед выполнением.

**Что делает:**
- Проверяет дневной лимит (50 лайков? 15 комментов? задаётся в конфиге аккаунта)
- Проверяет session лимит (не больше N действий за одну сессию)
- Считает consecutive failures **ПО ТИПУ ДЕЙСТВИЯ** — 2 подряд на один тип = этот тип заблокирован, остальные работают
- Проверяет cooldown между однотипными действиями (не 2 лайка за 3 секунды)
- Логирует каждое действие в SQLite с timestamp
- Считает session efficiency: (успешных действий / общее время) — записывает в outcomes

**Логика:**
```
agent хочет: like(post_id=123)
  → guard: лимит лайков сегодня? 47/50 — ок
  → guard: последний лайк 8 секунд назад? минимум 15 — WAIT 7 sec
  → guard: consecutive failures для like? 0 — ок
  → sdk.like_post(123)
  → результат: success
  → guard: reset failure counter для like

agent хочет: comment(post_id=456, text="...")
  → guard: лимит комментов? 15/15 — BLOCKED, лимит исчерпан
  → agent получает: ActionDenied("comment limit reached")
  → agent выбирает другое действие или завершает сессию
```

**Circuit breaker — по типу действия, не глобальный:**
```
create_post → fail → post_failures = 1
create_post → fail → post_failures = 2 → POST CIRCUIT OPEN
  → guard блокирует все post-действия
  → guard НЕ блокирует comment, like, follow — они работают через другой DOM
  → запись: "post circuit open: ProseMirror timeout x2"

comment → fail → comment_failures = 1
comment → success → comment_failures = 0 (reset)

Глобальный стоп: 3 РАЗНЫХ типа действий в circuit open → сессия завершена
  → "session terminated: 3+ action types broken, likely browser state issue"
```

**Это решает Проблему 3 и 4:** один сломанный тип не тянет за собой всю сессию. Агент продолжает комментить и лайкать даже если постинг сломался.

**Fallback-цепочка (код, не промпт):**
```
create_post failed? → guard предлагает: quote_repost (другой DOM)
quote_repost failed? → guard предлагает: comment (ещё другой DOM)
comment failed? → всё, этот пост пропускаем
```

### session.py — LiveSession

Сейчас каждое действие = connect → navigate → act → disconnect. Это детектируемый паттерн: пользователь "телепортируется" между страницами.

**Что делает:**
- Браузер открывается ОДИН раз в начале сессии
- Между действиями браузер остаётся открытым
- Навигация через "скролл ленты" или клик, не через прямые URL
- Disconnect только в конце сессии

**Жизненный цикл:**
```
session.start(profile_id)
  → AdsPower открывает браузер
  → Playwright подключается через CDP
  → Браузер на главной Binance Square

session.scroll_feed()
  → behavior.py: скролл вниз, пауза, скролл, остановка на посте

session.execute(action)
  → действие выполняется в контексте текущей страницы
  → если нужна навигация — через клик, не через URL

session.end()
  → disconnect
```

### behavior.py — HumanBehavior

Антидетект на уровне поведения. AdsPower даёт fingerprint (canvas, WebGL, fonts) + прокси (разные IP на каждый профиль). Но поведение тоже паттерн.

**Что делает:**
- Задержки между действиями: не фиксированные 30 сек, а распределение (15-90 сек, weighted random — чаще 20-40, реже 60-90)
- Скролл: не мгновенный jump, а серия маленьких скроллов с паузами
- Порядок действий: shuffle в рамках одного приоритета (не "все лайки, потом все комменты")
- Случайные "холостые" действия: проскроллить мимо поста, задержаться на картинке, вернуться назад
- Вариация между сессиями: разная длительность, разное количество действий
- Jitter на расписание: сессия стартует не ровно в 08:00, а в 07:47-08:13 (random offset)

**Пример сессии с behavior:**
```
[0:00] открыл ленту
[0:12] скролл вниз (3 поста)
[0:18] остановился на посте, "читает" 8 секунд
[0:26] лайк
[0:34] скролл вниз (2 поста)
[0:51] остановился, "читает" 12 секунд
[1:03] начал писать коммент
[1:47] отправил коммент
[1:55] скролл вниз (4 поста) — пропустил без действий
[2:23] лайк
[2:31] скролл вверх (вернулся к посту)
[2:45] скролл вниз, ушёл дальше
...
```

Не бот который делает action-pause-action-pause. Человек который листает ленту.

## Слой 2: Metrics — глаза агента

**Принцип:** каждое действие получает отложенную оценку. Без метрик агент слепой — его стратегия это галлюцинация.

### collector.py — MetricsCollector

**Что делает:**
- Запускается **отдельно от сессии агента** (фоновый процесс или отдельный cron)
- Проходит по таблице actions (действия агента)
- Для каждого действия старше 6 часов, у которого нет metrics — собирает результат

**Сбор по типу действия:**

| Действие | Что собирать через 6h | Что собирать через 24h |
|----------|----------------------|----------------------|
| post | views, likes, comments, quotes | views, likes, comments |
| comment | likes на коммент, reply от автора (да/нет), reply от других | likes на коммент, новые replies |
| like | — | — (лайки не дают измеримых метрик) |
| follow | follow_back (да/нет) | — |
| quote_repost | views, likes, comments | views, likes, comments |

**Как собирает:**
- Посты: `sdk.get_post_stats(post_id)` — views, likes, comments
- Комменты: `sdk.get_post_comments(post_id)` — найти свой коммент, есть ли reply, сколько likes
- Профиль: `sdk.get_my_stats()` — общий snapshot followers (для тренда, не для атрибуции)

### scorer.py — ActionScorer

**Что делает:**
- Берёт action + outcome из SQLite
- Считает engagement score для каждого действия
- Агрегирует по категориям: тип контента, целевой автор, время суток, тема
- **Генерирует автоматические уроки** из паттернов в данных

**Оценка на основе engagement (не followers_delta):**

followers_delta нельзя атрибутировать к конкретному действию — если за день было 3 коммента и 1 пост, невозможно понять что из этого принесло фолловеров. Поэтому scorer использует **прямые метрики действия**:

Для поста:
```
post_score = (views * 0.1) + (likes * 2) + (comments * 5) + (quotes * 8)
```

Для комментария:
```
comment_score = (author_replied * 20) + (likes_on_comment * 3) + (other_replies * 5)
```

author_replied = 20 баллов, потому что это самый сильный сигнал: автор заметил, его аудитория видит тред.

**Агрегация — вот где ценность:**
```
По автору (relationships):
  TMM Crypto (39K): 5 comments, avg_score: 8.4, author_replied: 2/5 (40%)
  Random Author X (2K): 3 comments, avg_score: 1.2, author_replied: 0/3

По типу контента (performance):
  BTC analysis + chart: avg_views: 89, avg_likes: 3.2
  market mood + meme: avg_views: 145, avg_likes: 5.8
  ETH analysis: avg_views: 34, avg_likes: 0.8

По времени:
  08:00-12:00 UTC: avg_comment_score: 6.2
  18:00-22:00 UTC: avg_comment_score: 3.1
  22:00-02:00 UTC: avg_comment_score: 0.8
```

Followers trend трекается отдельно на уровне профиля (ежедневный snapshot) — для общей картины "расту / стагнирую / падаю", но не для оценки конкретных действий.

**Автогенерация уроков:**
```python
def generate_lessons(insights):
    lessons = []

    # Сравнение контента с/без картинки
    with_image = get_insight("content_type", "with_image")
    without_image = get_insight("content_type", "without_image")
    if with_image.avg_score > without_image.avg_score * 2 and with_image.sample_count >= 3:
        lessons.append(f"posts without images get {ratio}x less engagement — always attach image")

    # Лучшее время для комментов
    best_hour = max(hour_insights, key=lambda h: h.avg_score)
    worst_hour = min(hour_insights, key=lambda h: h.avg_score)
    if best_hour.avg_score > worst_hour.avg_score * 2 and best_hour.sample_count >= 3:
        lessons.append(f"comments at {best_hour}:00 UTC perform {ratio}x better than {worst_hour}:00")

    # Авторы которые никогда не отвечают
    for author in author_insights:
        if author.sample_count >= 3 and author.author_reply_rate == 0:
            lessons.append(f"{author.name} never replies to comments — stop targeting")

    return lessons
```

### store.py — MetricsStore

SQLite таблицы:

```sql
-- Каждое действие агента
CREATE TABLE actions (
    id INTEGER PRIMARY KEY,
    agent_id TEXT NOT NULL,
    session_id TEXT NOT NULL,
    action_type TEXT NOT NULL,      -- post, comment, like, follow, quote_repost
    target_id TEXT,                  -- post_id или user_id
    target_author TEXT,              -- кому принадлежит пост
    target_author_followers INTEGER, -- сколько фолловеров у автора на момент действия
    topic TEXT,                      -- о чём (BTC, ETH, market mood)
    content_type TEXT,               -- analysis, meme, hot_take, question
    has_image BOOLEAN DEFAULT FALSE, -- пост/коммент с картинкой или без
    timestamp_utc TEXT NOT NULL,
    success BOOLEAN NOT NULL,
    error_message TEXT
);

-- Отложенные метрики (собирает collector)
CREATE TABLE outcomes (
    action_id INTEGER REFERENCES actions(id),
    collected_at TEXT NOT NULL,
    hours_after INTEGER NOT NULL,   -- 6 или 24
    views INTEGER,
    likes INTEGER,
    comments INTEGER,
    author_replied BOOLEAN,
    other_replies INTEGER,
    score REAL                      -- вычисленный scorer
);

-- Агрегированные инсайты (обновляет scorer)
CREATE TABLE insights (
    agent_id TEXT NOT NULL,
    dimension TEXT NOT NULL,         -- "author", "content_type", "hour", "topic", "has_image"
    dimension_value TEXT NOT NULL,   -- "TMM Crypto", "meme", "10", "BTC", "true"
    sample_count INTEGER,
    avg_score REAL,
    author_reply_rate REAL,         -- для dimension=author
    last_updated TEXT
);

-- Ежедневный snapshot профиля (для тренда роста)
CREATE TABLE profile_snapshots (
    agent_id TEXT NOT NULL,
    date TEXT NOT NULL,
    followers INTEGER,
    following INTEGER,
    total_views INTEGER,
    total_likes INTEGER,
    PRIMARY KEY (agent_id, date)
);

-- Метрики эффективности сессий
CREATE TABLE session_stats (
    session_id TEXT PRIMARY KEY,
    agent_id TEXT NOT NULL,
    started_at TEXT NOT NULL,
    ended_at TEXT NOT NULL,
    duration_seconds INTEGER,
    planned_actions INTEGER,        -- сколько было в плане
    executed_actions INTEGER,       -- сколько выполнено
    successful_actions INTEGER,     -- сколько успешно
    failed_actions INTEGER,
    circuits_opened TEXT,           -- JSON: ["post", "comment"] — какие типы сломались
    efficiency REAL                 -- successful / duration_minutes
);
```

session_stats решает **Проблему 5**: journal больше не может врать. Реальные цифры: сколько планировали, сколько сделали, сколько времени потратили, какие типы действий сломались.

## Слой 3: Memory — опыт агента

**Принцип из поста Meta Alchemist:** память без фильтра качества — это лог. Тест перед сохранением: "агенту это полезно в следующей сессии?"

### Структура памяти агента

```
agents/aisama/
  identity.md         # Кто он. Не меняется (или меняется редко, вручную)
  goal.md             # Цель. Эволюционирует по мере роста (0→100→1K→10K followers)
  strategy.md         # Генерируется analyst.py на основе данных. Не вручную
  lessons.md          # Одна строка = один урок. Самый ценный файл
  relationships.md    # Граф: автор → history → score → тактика
  performance.md      # Что работает: тип контента → метрики
  tactics.md          # Тактики с Boundary/Contradiction
  journal.md          # Последние 5 сессий. Старое сжимается compactor-ом
  archive/            # Сжатые старые сессии (агент не читает)
```

### lessons.md — автоматические + ручные уроки

**Два источника:**

1. **scorer.py генерирует автоматически** из данных (код, детерминированно):
```
- posts without images get 7x less engagement — always attach image [auto, n=6, confidence: high]
- comments after 22:00 UTC get no author replies — avoid late sessions [auto, n=8, confidence: high]
- ETH analysis posts underperform BTC by 3x — reduce ETH content [auto, n=4, confidence: medium]
- comments on authors 10-50K followers get 5x more author replies than 100K+ [auto, n=12, confidence: high]
- CryptoAnalyst_42 never replies to comments — stop targeting [auto, n=3, confidence: medium]
```

2. **LLM (reviewer.py) добавляет качественные наблюдения** (то что scorer не видит):
```
- ProseMirror editor times out when browser is cold — retry once with 5s pause [manual, session 3]
- SIGN token spam floods feed periodically — skip posts mentioning SIGN [manual, session 2]
- TMM Crypto values transparency — mention my own wrong calls to build rapport [manual, session 3]
- when recommended feed is garbage (all 0-like posts) — switch to "For You" or search by $CASHTAG [manual, session 3]
```

### relationships.md — граф связей

Генерируется из insights таблицы (dimension="author"). Агент читает это для выбора целей:

## TMM Crypto

### Тактики с Boundary/Contradiction (из поста Meta Alchemist)

Каждая тактика имеет не только "что работает", но и "когда перестаёт" и "где противоречит":

```markdown
## Тактика: комменты на инфлюенсеров

Claim: комменты на постах авторов с 10-50K фолловерами дают лучшую конверсию

Mechanism: у таких авторов достаточно аудитории чтобы тебя заметили,
но недостаточно комментов чтобы твой потерялся

Boundary: на авторах 100K+ коммент тонет в сотнях других — ROI падает
Boundary: 3+ комментов одному автору за сессию выглядит как stalking
Boundary: если автор никогда не отвечает на комменты (0/3+) — переключись

Contradiction: иногда 500K автор отвечает на коммент — это даёт x10 эффект.
Не игнорируй крупных полностью, но не делай их основной целью

Evidence: n=12 interactions, confidence: high [auto-updated by scorer]
```

Это даёт агенту **нюанс**: не "всегда комменть инфлюенсеров", а "комменть 10-50K, на 100K+ только если пост свежий и мало комментов, не больше 2 на одного за сессию".

### compactor.py — сжатие журнала

**Проблема:** journal.md растёт на ~30 строк за сессию. К сессии 100 = 3000 строк.

**Решение — два процесса:**

**Процесс 1: Детерминированный (код).** Обновление relationships.md и performance.md из таблиц insights и outcomes:
```
compactor_data():
  → SELECT из insights таблицы → генерит performance.md (таблицы с метриками)
  → SELECT из insights WHERE dimension='author' → генерит relationships.md (граф связей)
  → SELECT из session_stats → вырезает sessions старше 5 из journal.md → перемещает в archive/
```
Это чистый код — читает SQLite, пишет markdown. Без LLM. Детерминированно.

**Процесс 2: Качественный (LLM через reviewer.py).** Добавление нюансированных наблюдений:
```
reviewer():
  → читает journal сессии
  → добавляет 0-2 строки в lessons.md (только если узнал что-то новое)
  → обновляет boundary/contradiction в тактиках если данные противоречат
```
Это LLM — вероятностный. Но его вклад ограничен: максимум 2 строки в lessons, и только качественные наблюдения которые scorer не может извлечь из данных.

**Разделение чётко по иерархии надёжности:** количественное (performance, relationships) = код. Качественное (lessons, tactics nuances) = LLM.

## Слой 4: Strategy — мозг агента (LLM)

**Принцип:** LLM отвечает ТОЛЬКО за стратегию и креатив. Не за лимиты, не за паузы, не за подсчёт метрик. Его задача: "что делать и почему".

### analyst.py — анализ ситуации

Запускается в начале сессии. LLM получает:
- performance.md (что работает — сгенерировано кодом из данных)
- relationships.md (с кем работать — сгенерировано кодом из данных)
- lessons.md (что не повторять — авто + ручные)
- tactics.md (тактики с boundary/contradiction)
- текущие метрики агента (followers, views trend — из profile_snapshots)
- текущий рынок (цены, тренды, новости — через SDK data methods)

Выдаёт обновлённый strategy.md:
```markdown
## Текущая фаза: Early Growth (4 → 50 followers)

## Фокус этой недели
- 70% comments на авторов 10-50K (основной драйвер: author_reply_rate 35%)
- 20% posts с картинками (meme/mood: 145 avg views vs 34 у ETH analysis)
- 10% reply-to-replies (поддержание связей с TMM Crypto и новыми)

## Конкретные решения
- Прекратить ETH analysis (avg views 34, score 5.1 — худший тип контента)
- Увеличить долю news hot takes (score 29.0, но n=1 — протестировать ещё)
- TMM Crypto: max 1 коммент за сессию (boundary: stalking risk)
- Прекратить CryptoAnalyst_42 (0% reply rate за 3 коммента)
- Найти 2-3 новых автора в диапазоне 15-30K

## Метрика успеха этой недели
- +15 followers (trend: +4 за первые 2 дня)
- 2 новых автора с reply_rate > 0
```

### planner.py — план сессии

**Решает Проблему 6:** конкретный deliverable list ДО выхода в ленту.

LLM получает: strategy.md + текущий фид (предварительно отфильтрованный) + рынок + новости.

**Предварительная фильтрация фида (код, не LLM):**
```python
def filter_feed(posts):
    """Код убирает мусор ДО того как LLM увидит фид"""
    return [p for p in posts
            if p.likes >= 5                    # минимальный engagement
            and p.text_length > 50             # не спам
            and not is_spam(p.text)            # не SIGN token promo
            and p.author_followers >= 1000]    # минимальная аудитория автора
```

**Это решает Проблему 1:** агент не листает мусорный фид. Planner получает только посты которые стоят внимания.

**Fallback когда весь фид мусор:**
```
filtered_feed пустой?
  → переключить tab: "For You" → "Trending" → search by $BTC/$ETH
  → если всё пусто: сессия = "только посты" (писать свой контент без комментирования)
```

Planner выдаёт конкретный план:
```json
[
  {"action": "check_replies", "priority": 0, "reason": "всегда первое действие"},
  {"action": "comment", "target": "post_384772", "author": "CryptoWhale (28K)",
   "priority": 1, "reason": "10-50K автор, пост свежий (2h), 12 лайков, тема SOL",
   "fallback": "like only if comment fails"},
  {"action": "like", "target": "post_384772", "priority": 1},
  {"action": "post", "topic": "SOL MACD cross", "content_type": "analysis",
   "priority": 2, "reason": "SOL gap в моём контенте, свежие данные",
   "image": "screenshot_chart(SOL, 4h)",
   "fallback": "post without chart if screenshot fails"},
  {"action": "comment", "target": "post_384800", "author": "DeFi_Mike (15K)",
   "priority": 3, "reason": "новый автор, тестируем reply rate"}
]
```

**Каждое действие имеет fallback** — это решает **Проблему 2**: если картинка не получилась, пост выходит без неё. Если коммент не получился — хотя бы лайк.

### reviewer.py — оценка после сессии

Две части:

**Часть 1: Детерминированная (код).** Записывает session_stats:
```python
session_stats = {
    "planned_actions": len(plan),
    "executed_actions": executed_count,
    "successful_actions": success_count,
    "failed_actions": fail_count,
    "circuits_opened": ["post"],  # какие типы сломались
    "duration_seconds": end - start,
    "efficiency": success_count / (duration_minutes or 1)
}
# → INSERT INTO session_stats
```

**Часть 2: Качественная (LLM).** Смотрит на план vs результат:
- Добавляет 0-2 строки в lessons.md (только НОВЫЕ наблюдения)
- Обновляет journal.md (краткое описание сессии)
- НЕ оценивает "была ли сессия успешной" — это делает код через session_stats и scorer

**Ограничение reviewer:** LLM не видит свои собственные метрики как "успех/провал". Код считает efficiency и scores. LLM только добавляет качественные наблюдения которые код не может извлечь (e.g., "фид был загрязнён спамом нового токена").

### Bootstrap-стратегия (первые 15 сессий)

Первые 2 недели данных мало — performance.md и relationships.md почти пустые. Агент не может оптимизировать то, что не измерил.

**Решение: хардкодированная exploration-стратегия для bootstrap:**

```markdown
## Bootstrap Strategy (sessions 1-15)

Цель: собрать данные по всем измерениям, НЕ оптимизировать.

Обязательный микс каждой сессии:
- 1 пост с картинкой (чередовать типы: analysis, meme, hot take, news)
- 3-4 коммента (чередовать: авторы разного размера, разные темы)
- 2-3 лайка
- 1 follow (если есть подходящий автор)

Чередование обязательно: НЕ два поста одного типа подряд, НЕ три коммента на авторов одного размера.

Время сессий: варьировать (утро, день, вечер) — чтобы собрать данные по временным слотам.

Переход в адаптивный режим: когда sample_count >= 3 по каждому content_type И по каждому author_bracket.
```

analyst.py проверяет `MIN(sample_count) FROM insights` — если < 3 по любому измерению, возвращает bootstrap-стратегию вместо оптимизированной.

## Полный цикл одной сессии

```
[T-30min] METRICS COLLECTOR (фоновый cron)
  → собирает outcomes за прошлые действия (views, likes, replies)
  → scorer считает scores
  → scorer генерит автоматические lessons если нашёл паттерн (n >= 3)
  → обновляет insights таблицу
  → делает profile_snapshot (followers, views)

[T-5min] MEMORY COMPACTOR (если journal > 5 сессий)
  → compactor_data(): обновляет performance.md и relationships.md из insights (код)
  → вырезает старые сессии из journal → archive/

[T=0] SCHEDULER запускает сессию (с jitter ±15 min)

[T+0:00] ANALYST (LLM)
  → проверяет: bootstrap или адаптивный режим?
  → читает: lessons.md, relationships.md, performance.md, tactics.md
  → читает: рынок, новости, тренды (через SDK data methods)
  → обновляет strategy.md

[T+0:01] PLANNER (LLM)
  → читает: strategy.md + текущий фид (предварительно отфильтрованный кодом)
  → если filtered_feed пустой → fallback на другие табы / search
  → генерит план: [{action, target, priority, reason, fallback}, ...]

[T+0:02] LIVE SESSION открывает браузер
  → behavior.py: "скроллит" ленту естественно

[T+0:02 — T+0:30] EXECUTION LOOP
  для каждого действия из плана:
    → guard.py: можно? (лимит, cooldown, circuit breaker ПО ТИПУ)
    → если circuit open для этого типа → попробовать fallback
    → behavior.py: задержка, скролл к посту, "чтение"
    → sdk: выполнение
    → guard.py: результат (success/fail → failure counter ПО ТИПУ)
    → store.py: запись в actions таблицу

[T+0:30] REVIEWER
  → код: записать session_stats (planned, executed, success, failed, efficiency)
  → LLM: 0-2 строки в lessons.md + запись в journal.md

[T+0:31] DISCONNECT
  → браузер закрывается
  → сессия завершена
```

## Что взято из поста Meta Alchemist

### 1. Фильтр качества памяти
Тест: "агенту это полезно в следующей сессии?"
- **Хранить:** инсайты из данных, уроки из ошибок, эффективные тактики с границами
- **Не хранить:** "лайкнул пост #12345", "скролл занял 12 секунд", "использовал screenshot_chart()"
Реализация: compactor_data() обновляет md из SQLite. Телеметрия остаётся в SQLite, знания — в md.

### 2. Boundary / Contradiction для тактик
Каждая тактика имеет не только "что работает", но и "когда перестаёт работать" и "где противоречит другим знаниям".
Реализация: tactics.md с полями Claim/Mechanism/Boundary/Contradiction. Evidence обновляется scorer.

### 3. lessons.md — однострочные уроки
Самый быстрочитаемый файл. Два источника: автоматические (scorer.py — код) и ручные (reviewer.py — LLM).
Каждый урок помечен: [auto, n=X, confidence: Y] или [manual, session N].

### 4. Разделение телеметрии и знаний
Телеметрия (action logs, timings, error counts) → SQLite, для diagnostics и scorer.
Знания (что работает, с кем, когда) → md-файлы, для стратегии.
Агент читает только md. Код обрабатывает SQLite и генерит md.

### Что НЕ взято (и почему)
- **UCB1 Multi-Armed Bandit** — требует сотен итераций. scorer.py с простой агрегацией работает лучше на малых данных (5-10 сессий в день)
- **Promotion ladder (5 уровней)** — избыточно. Достаточно: confidence: high/medium/low на основе sample_count
- **Doctrine synthesis через LLM** — агент не исследователь. Конкретные метрики + конкретные уроки > мета-доктрины
- **Obsidian vault как UI** — агенты не люди. SQLite + md-файлы которые оператор может открыть при желании

## Иерархия надёжности (из стандарта разработки)

Применена к этой архитектуре:

| Уровень | Компонент | Тип | Что делает |
|---------|-----------|-----|------------|
| 1 (max) | guard.py | Код | Лимиты, circuit breaker по типам, cooldown. Нельзя обойти |
| 2 | scorer.py, compactor_data() | Код | Метрики, агрегация, автоуроки, генерация performance.md/relationships.md |
| 3 | filter_feed(), session_stats | Код | Фильтрация мусора, реальный КПД сессии |
| 4 | analyst.py, planner.py | LLM | Стратегия, план сессии — вероятностно, но на основе данных от слоёв 1-3 |
| 5 | reviewer.py | LLM | Качественные lessons — max 2 строки за сессию, ограниченный scope |
| 6 | identity.md, промпт | Текст | Персона, стиль — только намерения |

LLM делает то, что только LLM может: стратегию, креатив, понимание контекста. Всё остальное — код.

## Мониторинг для оператора

Оператор не вмешивается, но должен иметь возможность **быстро понять что происходит**.

**Способ 1: session_stats в SQLite.** Простой SELECT показывает:
```sql
SELECT agent_id, date(started_at),
       planned_actions, successful_actions, efficiency,
       circuits_opened
FROM session_stats
ORDER BY started_at DESC
LIMIT 20;
```

**Способ 2: Алерты (опционально, фаза 2).** Скрипт проверяет:
- efficiency < 0.3 три сессии подряд → "agent X degrading"
- followers_trend отрицательный 7 дней → "agent X losing followers"
- circuits_opened содержит 3+ типов → "agent X browser issues"

**Способ 3: Просмотр md-файлов агента.** lessons.md, strategy.md, performance.md — всё читаемо.
```
