## Раздел A: Эскиз продукта (идея)

### A.1. Название продукта
Binance Square Toolkit

### A.2. Одно предложение — что это
SDK/тулкит для автоматизированной публикации контента и engagement на Binance Square, управляемый AI-агентом, монетизация через Write to Earn.

### A.3. Проблема, которую решаем
У меня 6 аккаунтов Binance Square с разными нишами (DeFi, трейдинг, новости, инфраструктура, образование, институционалы). Каждому нужны 3-5 постов/день плюс лайки и комменты для наращивания аудитории и W2E-комиссий. Это 18-30 постов, 180-360 лайков и 70-140 комментов ежедневно — вручную невозможно. Без автоматизации аккаунты простаивают, W2E-доход теряется.

### A.4. Кто пользователь
Personal project. AI-агент (Claude/Codex) — непосредственный потребитель API тулкита: он вызывает функции для парсинга, генерации, публикации и engagement.

### A.5. Что продукт ДОЛЖЕН делать (обязательно)
1. Парсить ленту Binance Square, трендовые статьи, индекс страха/жадности, горячие хэштеги и рыночные данные через bapi-эндпоинты
2. Ранжировать трендовые темы по engagement-скору (просмотры, лайки, комменты)
3. Генерировать текстовый контент через AI (Claude/OpenAI) — `ContentGenerator` и `CommentGenerator` как инструменты агента — со стилем персоны, реальными рыночными данными, трендовыми кэштегами/хэштегами
4. Публиковать посты на Binance Square через Playwright CDP (требуются client-side подписи)
5. Лайкать посты через httpx (bapi-эндпоинт)
6. Комментировать посты через Playwright CDP (DOM-based input)
7. Подписываться на авторов через Playwright CDP
8. Управлять конфигурациями 6+ аккаунтов через YAML (персона, прокси, лимиты)
9. Обеспечивать дневные лимиты на аккаунт (посты: 3-5, лайки: 30-60, комменты: 12-24)
10. Логировать все действия в SQLite для трекинга и отладки
11. Захватывать и управлять bapi credentials (cookies + headers) через CDP
12. Подключаться к профилям AdsPower для антидетект-браузинга

### A.6. Что продукт МОЖЕТ делать (желательно)
1. Генерация медиа-контента (ценовые графики, HTML-шаблоны → PNG)
2. Создание полных статей (не только коротких постов)
3. Аналитический дашборд engagement по аккаунтам/темам
4. A/B тестирование разных стилей контента
5. Автоподстройка частоты постинга по данным engagement
6. Трекинг дохода (W2E-комиссия по аккаунтам)
7. Просмотр профилей авторов и скрейпинг истории постов
8. Удаление постов/комментов

### A.7. Что продукт НЕ ДОЛЖЕН делать (границы)
1. Не принимать автономных решений — все решения от AI-агента, который получает задания от человека
2. Не хранить API-ключи в коде или конфигах (только .env)
3. Не взаимодействовать между своими аккаунтами (без само-лайков, само-комментов, само-подписок)
4. Не превышать неявные лимиты Binance (консервативный подход)
5. Не публиковать контент без стиля персоны и реальных данных
6. Не работать без валидных credentials (fail fast при истёкших cookies)

## Раздел B: Технические ограничения

### B.1. Платформа
- [x] Скрипт / CLI
- Python-тулкит, потребляемый AI-агентом. Без GUI. Без веб-интерфейса.

### B.2. Язык и фреймворк
- Python 3.12
- httpx (async HTTP-клиент для bapi парсинга + лайков)
- Playwright (CDP-подключение к браузерам AdsPower для постинга, комментирования, подписок)
- anthropic / openai SDK (генерация контента — инструменты агента)
- APScheduler (планирование задач — опционально, агент может управлять циклами сам)
- pydantic (валидация конфигов)
- aiosqlite (async SQLite)
- pyyaml (загрузка конфигов)

### B.3. База данных
- [x] Нужна (данные сохраняются между сессиями)
- SQLite с WAL mode. 7 таблиц: credentials, actions_log, daily_stats, content_queue, parsed_trends, parsed_posts, discovered_endpoints.
- YAML — источник истины для конфигов аккаунтов/персон. SQLite хранит только runtime-данные.

### B.4. Внешние сервисы и API
- Binance bapi (internal endpoints) — парсинг ленты, лайки. Требуются захваченные cookies + 10 специфических заголовков.
- Binance public market API — цены монет, объём, изменение за 24ч. Без аутентификации.
- Claude API (Anthropic) — генерация контента (инструмент агента). Auth: ANTHROPIC_API_KEY.
- OpenAI API — генерация контента (fallback). Auth: OPENAI_API_KEY.
- AdsPower local API (localhost:50325) — запуск/остановка профилей браузера. Без внешней аутентификации.
- iProxy — мобильные прокси, одно устройство на аккаунт.

### B.5. Где запускается
- [x] На моём компьютере (Windows 11)
- Локальное выполнение. AdsPower работает локально. Облачное развёртывание не планируется.

### B.6. Ожидаемый размер
- [x] Средний (2000-10000 строк, 5-10 модулей)
- Модули: parser, content, activity, accounts, session, bapi, db, scheduler. Плюс конфиги и тесты.

## Раздел C: Архитектурные требования

### C.1. Ссылка на технические требования
Проект разрабатывается в соответствии с:
- 01_technical_requirements.md (все стандарты обязательны)

### C.2. Дополнительные ограничения проекта
- Все браузерные действия через профили AdsPower (никогда не запускать raw Playwright)
- Захват credentials — отдельный явный шаг (не автоматический при каждом действии)
- Каждый модуль тестируем без live-подключения к Binance (mock-friendly интерфейсы)
- Валидация конфигов при запуске — fail fast если YAML-конфиги невалидны

### C.3. Требования безопасности
- [x] Работа с токенами (credentials из браузера — cookies, csrftoken, fvideo-id, fvideo-token)
- [x] Работа с крипто (платформа Binance, монетизация W2E)
- API-ключи только в .env, никогда в YAML или коде
- Захваченные credentials в SQLite (локальный файл, не передаётся)
- Credentials каждого аккаунта изолированы (без кросс-использования)

### C.4. Требования надёжности
- [x] Важно — логировать ошибки и продолжать работу
- Если credentials одного аккаунта истекли, остальные продолжают работать
- Если AI API упал, пропустить цикл генерации и повторить в следующем
- Если профиль AdsPower не запустился, залогировать ошибку, пропустить аккаунт, продолжить с остальными
- Все действия обёрнуты в try/except со структурированным логированием
- Graceful shutdown: завершить текущее действие, не начинать новые

## Раздел D: Этапы реализации

### D.1. Разбивка по этапам

**Этап 1: Фундамент (Парсер + БД + Аккаунты)**
После этого этапа: можно парсить ленту Binance Square, трендовые статьи, fear & greed, горячие хэштеги и рыночные данные. Данные в SQLite. Конфиги аккаунтов загружаются из YAML. Захват credentials работает.

**Этап 2: Контент-движок (Генерация + Публикация)**
После этого этапа: можно генерировать текстовый контент через AI (агент вызывает `ContentGenerator` как инструмент) со стилями персон и публиковать посты на Binance Square через Playwright CDP. Очередь контента в SQLite.

**Этап 3: Движок активности (Лайки + Комменты + Подписки)**
После этого этапа: можно лайкать посты через httpx, комментировать и подписываться через Playwright CDP. Лимиты действуют. Все действия логируются.

**Этап 4: Оркестрация (Планировщик + Мульти-аккаунт)**
После этого этапа: APScheduler (или агент напрямую) запускает циклы парсинг→генерация→публикация→engagement по всем 6 аккаунтам. Правила антидетекта соблюдаются (без пересечений, без кросс-взаимодействий).

**Этап 5: Медиа + Аналитика (Фаза 2-3)**
После этого этапа: генерация медиа-контента (графики, картинки), аналитический дашборд, A/B тестирование, трекинг дохода.

### D.2. Спецификации модулей
Перед реализацией каждого этапа создать документы спецификаций (docs/spec_[module].md) с:
1. User Stories
2. Модель данных
3. API (сигнатуры функций, входы/выходы, обработка ошибок)
4. Бизнес-логика (лимиты, валидация, переходы состояний)
5. Граничные случаи (истёкшие credentials, падения API, конкурентный доступ)

Примечание: без UI-компонентов — это headless-тулкит.

### D.3. Приоритеты
1. Парсер (фундамент — всё зависит от данных трендов)
2. Менеджер сессий + захват credentials (нужен и для httpx, и для CDP)
3. Контент-движок (ключевая ценность — генерация и публикация постов)
4. Движок активности (engagement — лайки, комменты, подписки)
5. Мульти-аккаунтная оркестрация (масштабирование с 1 до 6 аккаунтов)
6. Генерация медиа (желательно, Фаза 2)
7. Аналитика (оптимизация, Фаза 3)

### D.4. Критерии "Готово" по этапам

**Этап 1:** Запустить парсер, увидеть 100 постов ленты + 100 статей + fear/greed + хэштеги в SQLite. Конфиг аккаунта загружается без ошибок. Захват credentials извлекает cookies и headers из профиля AdsPower.

**Этап 2:** Запустить генерацию контента для 1 аккаунта, увидеть AI-сгенерированный пост с реальными рыночными данными и стилем персоны. Пост появляется на Binance Square. Проверить в браузере.

**Этап 3:** Запустить цикл активности для 1 аккаунта, увидеть 5-10 лайков в actions_log, 2-4 опубликованных коммента (видны на Binance Square), 1-2 выполненных подписки.

**Этап 4:** Запустить планировщик (или агент запускает циклы), оставить на 24 часа. Проверить: 6 аккаунтов, каждый опубликовал 3-5 постов, лайкнул 30-60 постов, прокомментировал 12-24 раза. Без банов, без ошибок в логах.

**Этап 5:** Посты включают сгенерированные ценовые графики. Дашборд показывает метрики engagement по аккаунтам.

## Раздел E: Приёмка и контроль качества

### E.1. Как я проверяю работу
- Запустить тулкит, проверить Binance Square в браузере — посты появляются в профиле каждого аккаунта
- Проверить SQLite — actions_log содержит записи для каждого поста/лайка/коммента
- Проверить daily_stats — числа в ожидаемых диапазонах (3-5 постов, 30-60 лайков на аккаунт)
- Мониторить аккаунты 7 дней — без банов, без предупреждений от Binance
- Проверить качество контента — читается как написанное человеком, с реальными цифрами, соответствует стилю персоны

### E.2. Что считается багом
- Тулкит падает без понятного сообщения об ошибке
- Пост опубликован, но не виден на Binance Square
- Credentials истекли, а тулкит продолжает пытаться с мёртвыми credentials вместо fail fast
- Лимиты превышены (больше действий, чем сконфигурированный максимум)
- Обнаружено кросс-взаимодействие (аккаунт A лайкнул пост аккаунта B)
- Контент содержит AI-клише ("in the ever-evolving world of", "it's worth noting that")
- Действие залогировано как успешное, но фактически провалилось на стороне Binance

### E.3. Что считается успехом
- 6 аккаунтов работают по командам агента 7+ дней без банов
- Каждый аккаунт публикует 3-5 постов/день с уникальным стилем персоны и реальными рыночными данными
- Движок активности генерирует 30-60 лайков и 12-24 коммента на аккаунт в день
- Все действия логируются, дневная статистика агрегируется, ошибки отслеживаемы
- W2E-доход измерим (даже если изначально небольшой)
- Добавление 7-го аккаунта требует только новый YAML-конфиг и профиль AdsPower
